# Agent 项目优化建议（架构 & 工程层）

本文基于 **本仓库当前实现**（后端主会话、`ContextManager`、`动态工具`、`MAF` 等），从可维护性、可扩展性、稳定性与 Agent 演进能力给出方向；并与「可渐进落地」结合，避免与现状脱节的空泛排期。

**相关现有文档**：`docs/Agent概念与项目实现对照.md`、`docs/Agent优化.md`、`docs/动态工具与技能选择实现说明.md`、`docs/动态上下文压缩.md`。

---

## 一、Context 体系统一

### 项目现状

上下文确实多入口，但职责大致分明：

| 来源 | 作用 |
|------|------|
| `SessionState.History` | 持久对话 |
| `ContextManager.TrimHistory` | 轮数 + token 预算裁剪，且尊重 `ConversationCompactBoundary` |
| `ChatService.StreamPhases` Part1 | MAF `CompactionProvider`（工具折叠 → 摘要 → 截断） |
| `BuildHistoryForStreamingTurn` | **仅本轮流式**：在首条 system 上拼 `IdentitySuffix`、搜索抑制、固定 instruction，**不写回**持久历史 |
| `MessageAIContextProvider` | 记忆、知识库、计划、跨端待办等 **MAF 侧注入** |

本质仍在构造送入模型的消息集合；「多入口」带来的主要痛点是 **全链路可观测** 与 **改一处要摸多处**。

### 优化方向

抽象为可组合的 **Context 块**（概念即可，不必一上来定死接口名）：

- 块类型：历史 / 注入记忆 / 计划 / system 后缀 / 固定指令等
- 属性：是否仅本轮（ephemeral）、优先级、估算 token
- 统一 **Compose**：按优先级与预算合并，便于单点 dump（调试/Prompt Inspector）

### 落地建议

- **优先级：中高，但宜分阶段**：先增加「本轮送入模型前的只读快照」日志或调试 API（不改变行为），再逐步把 `BuildHistoryForStreamingTurn` 与块列表对齐；避免一次性大改 MAF `UseAIContextProviders` 与 Compaction 管线。
- **代码锚点**：`StreamChatTurnContext`、`ChatService.BuildHistoryForStreamingTurn`、`ChatTurnWorkflow`、`BuildContextProviders`。

## 二、Compaction（摘要链）智能化

### 项目现状

`ChatService.StreamPhases` 中 `BuildCompactionStrategy` 已为 **Pipeline**：`ToolResultCompactionStrategy` → `SummarizationCompactionStrategy` → `TruncationCompactionStrategy`，触发与 `historyBudget`、`SummarizationTriggerRatio` 等相关；摘要块带 `ConversationCompactBoundary` 标记，裁剪时受保护。

### 差距

- 策略主要按 **token 阈值**，对「当前用户句」的针对性有限
- 误删风险来自 **重要性未显式建模**，而非「完全没有摘要链」

### 优化方向：Query-aware Compaction（实验向）

- 引入对「当前轮用户文本」的相关性评分（可先启发式：关键词重叠；再考虑 embedding）
- 策略示意：高相关优先保留或整段保留；中相关摘要；低相关删除

### 落地建议

- **成本**：每轮 embedding 会增加延迟与费用，适合 **配置开关或实验分支**，并配合前后对比（质量、长度、失败率）
- **优先级：中长期研究项**，不宜与「上下文统一」并列为短期必做硬指标

## 三、动态工具系统

### 项目现状（已具备）

- Bootstrap + `search_available_tools` / `activate_tools`、`DynamicToolingTurnScope`、`MafMainSessionStreamRunner` 外层循环、同 pass 内刷新 Tools 等，见 `docs/动态工具与技能选择实现说明.md`。
- **`ToolCatalogIndex.Search` 已有关键词打分并按分数排序**，空 query 时有 pinned 逻辑；并非「完全无排序」。见 `backend/Services/DynamicTooling/ToolCatalogIndex.cs`。

### 可继续加强的方向

1. **排序信号扩展**：在现有 relevance 打分之上，叠加 **遥测**（成功率、延迟）、静态 **cost**（若可维护简单表），避免仅改启发式而无数据闭环。
2. **Tool Planning**：先输出结构化步骤再执行，可能减少 outer loop；实现成本高，需与动态工具、Verifier 协议对齐，适合 **专项迭代**。
3. **Tool Cache**：仅适合 **幂等、只读、可失效** 的场景（如某次 catalog 检索结果）；**当前文档 / 宿主文档** 类调用默认 **不建议** 长缓存，除非有明确版本或失效策略。

### 落地建议

- **优先级**：「增强排序与观测」为 **中**；Planning / 广谱 Cache 为 **中高难度、择期**。

## 四、Context Length：前置裁剪与预算

### 项目现状

- 发送前已有 `TrimHistory`、Compaction、`Reserved*Tokens`、`GetEffectiveMaxContextTokens` 等；**retry**（如 `ContextLengthRetryHelper` / `AgentRunMiddleware` 相关路径）更多是 **估算与上游限制不一致时的兜底**，并非「从不预裁剪」。
- 见 `ContextManager.cs`、`ChatService.StreamPhases.cs`。

### 优化方向

- **收紧估算或发送前再砍一刀**（在已有逻辑之上增量调整），降低触发 retry 的概率。
- **Token Budget Planner**：在现有 `ReservedSystemTokens` / `ReservedToolsTokens` / `ReservedOutputTokens` 思路上，用 **表格式文档** 固定「历史 / 注入 / 工具 schema / 输出」占比上限，避免再给用户堆大量细碎开关（符合产品「少开关」倾向）。

### 落地建议

- **优先级：中**；目标应是 **减少浪费性 retry**，而非承诺「零 retry」。

## 五、System Prompt 结构化

### 项目现状

- System 首条持久 + 每轮 `IdentitySuffix`、`ClientPageContextSuffixBuilder`、以及 `LatestIntent…` / 工具 schema 等拼进 **本轮** `HistoryToUse`，逻辑集中在 `BuildHistoryForStreamingTurn` 一带。

### 优化方向

按区块维护，例如：Identity、Capabilities、ClientContext、ToolInstructions、SafetyRules，再由 `SystemPromptBuilder.Build(...)` 合成字符串。

### 落地建议

- **优先级：高（相对容易、收益直观）**：重构以 **搬移与拼接** 为主，行为可测；便于按客户端 A/B 与维护。
- **代码锚点**：`ChatService.StreamPhases`、`ClientPageContextSuffixBuilder`、`BuildHistoryForStreamingTurn`。

---

## 六、Streaming 可观测性

### 项目现状

- WebSocket 已有 `stream_chunk`、`reasoning_chunk`、`tool_call_delta` 等；时间线侧有块协调（如 `Program.cs` 与 timeline 相关类型）。
- 业务侧已有 `agent_trace`、上下文类 trace 文案（如 `AgentTraceFormatter`）。

### 优化方向

- 引入 **`trace_id` / 轮次 id** 贯穿：模型调用、工具调用、Compaction 阶段，便于日志关联。
- 调试 UI（Timeline Debug）为 **增强项**，可在后端结构化事件就绪后再做前端。

### 落地建议

- **优先级：中高**：先 **结构化日志与 id**，再视需要做单页调试工具。

---

## 七、Memory / RAG

### 项目现状

- **双路径并存**：`MemoryContextProvider` 按用户句检索并 **注入**；插件 **`MemoryPlugin`** 已提供 `search_memory`、`save_memory`（见 `backend/Plugins/MemoryPlugin.cs`）。
- 知识库有 `KnowledgeBaseContextProvider`。

### 差距

- 不是「没有工具」，而是 **注入与主动搜索的分工**、是否重复、何时以何为准 需要产品/协议上更清晰。

### 优化方向

- 明确策略：例如默认轻量注入 + 强依赖时再强调调用 `search_memory`；或按记忆体量切换。
- 避免无节制双通道重复堆上下文。

### 落地建议

- **优先级：中**；以 **策略与文档** 为主，必要时再改默认 TopK 或门控。

---

## 八、Completion Verifier

### 项目现状

- 动态工具场景下有内置 `BuiltinTurnCompletionVerifier`，用于「假完成」等判断，与 `TurnRoute` 等配合。

### 优化方向

- Self-Reflection：显式检查「是否答完、是否漏工具、是否需再搜」等；可能带来 **额外模型调用或延迟**。

### 落地建议

- **优先级：中**；适合 **可选档位** 或轻量启发式，默认行为需谨慎，避免拖慢主路径。

---

## 九、Agent 状态机（中长期）

### 项目现状

- 已有 MAF `Workflow`（Part1 / Part2 / Tooling）与多处中间件；与「显式状态机」有重叠。

### 优化方向

- 显式状态：如 IDLE → PLANNING → TOOL_CALLING → RESPONDING → VERIFYING，利于复杂任务与排障。

### 落地建议

- **优先级：中长期**；需与 MAF 生命周期和 `maf-pragmatic` 约定一致，避免两套编排互相打架。

---

## 总结（修订后的优先级）

| 层级 | 建议 | 说明 |
|------|------|------|
| **近期可落地** | System Prompt 结构化；观测增强（trace/轮次 id、Compaction 阶段打点） | 改动面相对可控，直接利于维护与排障 |
| **中期** | Context 统一（先观测快照 → 再抽象 Compose）；Token 预算表与预裁剪微调；Memory 策略澄清 | 与现有 `Reserved*`、`TrimHistory` 演进衔接 |
| **研究与专项** | Query-aware Compaction；动态工具排序多维信号；Tool Planning | 成本与复杂度较高，应用数据与实验验证 |
| **中长期** | Agent 显式状态机；广谱 Self-Reflection；高风险 Tool Cache | 依赖架构稳定与明确场景 |

---

## 一句话总结

当前后端已是 **可运行的分层 Agent**（历史、压缩、动态工具、注入、流式、Verifier）；下一步优先把 **结构化 system、可关联的观测、与现状一致的文档** 做实，再按需投入 **智能压缩与更强工具编排**，避免脱离本仓库已有能力的重复建设。
