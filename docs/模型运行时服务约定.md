# 模型运行时服务约定

> **定位**：本文描述**本程序对「通过工具协议与后端交互的大模型」应承担的服务质量约定**——即运行时如何把模型当作一个需要**可靠环境反馈**的客户端来服务，而不是「教模型如何做人」的行为规范。  
> **实现以代码为准**；改工具管道、MAF、错误返回路径时，应核对是否仍满足本文。

---

## 1. 为什么要单独写

主会话里，模型通过 **function calling / tools** 与本机能力交互。若失败时静默、信息含糊、或环境与声明不一致，模型无法可靠自纠，用户会看到反复试错或幻觉性「已成功」。

其它产品里类似思想常散落在：**工具 Schema 与 Description**、**MCP 工具说明**、内部 *agent playbook* / *tool authoring guide*。本文把 **Taskly 后端对模型的义务**收拢为一处，便于评审与回归。

---

## 2. 核心原则

1. **环境诚实**：模型在对话里看到的「当前可用工具列表」应与**即将发出的推理请求**中携带的 `tools` 一致；激活、子集、bootstrap 等状态变更后，**同一轮 MAF 流**内须及时同步（见动态工具刷新逻辑）。
2. **失败可行动**：工具调用在绑定阶段或执行阶段失败时，应尽可能以 **tool 结果消息**的形式返回**具体原因**（缺哪个参数、类型不对、业务拒绝等），使模型能在下一轮修正参数或换策略；避免仅依赖「无正文的 HTTP 错误」或笼统一句「失败」。
3. **结构化、可扫描**：面向模型的错误文案宜有稳定前缀或格式（例如 `[工具调用失败]`、`[参数绑定失败]`），便于在上下文中识别，也与日志检索关键词对齐。
4. **安全与完整性的折中**：在**不泄露** API 密钥、敏感路径、个人隐私的前提下，尽量给出对排错有用的信息；**不把完整堆栈**塞进模型可见的 tool result（堆栈进日志）。
5. **与用户可见错误一致**：用户界面仍须遵守 [error-visibility](../.cursor/rules/error-visibility.mdc)（`message` 上屏等）；给模型的 tool result 可与 UI 文案分流，但**不要**在模型侧宣称成功而用户侧无说明。

---

## 3. 具体约定（服务端应做到）

### 3.1 工具列表与动态激活

- **`activate_tools` 成功后**：同一轮内发往模型的下一跳请求所使用的工具表，须包含新激活的工具（与 `DynamicToolingTurnState` / `ChatOptions.Tools` 刷新逻辑一致）。
- **工具名解析**：若上游使用限定名（如 `Plugin.func`），中间件应在进入安全与执行链前解析为注册所用的裸名与正确 `AIFunction`，避免「未找到函数」类假失败（见 `ToolQualifiedNameResolver`、`ToolInvocationMiddleware`）。

### 3.2 工具失败如何回到模型

- **JSON 参数与类型不符**：捕获 `JsonException`，返回说明性字符串（含工具名与修正提示），**不抛**出中断整轮对话（除非上层另有统一策略）。
- **绑定 / 必填参数缺失等**：`ArgumentException`、`InvalidOperationException` 等应格式化为带插件名、函数名与**截断后的** `Message` 的短文案返回给模型（见 `ToolInvocationFailureFormatter`），并打 **Warning 级日志**保留完整异常。
- **一般异常**：返回 `[工具调用失败] …` + `ErrorMessageHelper` 友好句；同样记录日志。**取消**：仅在协作 `CancellationToken` 已请求取消时，对 `OperationCanceledException` **继续向上抛出**，不把取消误报成业务失败。

### 3.3 工具实现侧

- 插件方法返回字符串时，失败分支应包含**可转述原因**（与 [error-visibility](../.cursor/rules/error-visibility.mdc) 一致），便于模型复述给用户。
- 对易被模型漏填的参数，可在 **Schema 层**用默认值降低绑定失败率（与「仍返回明确错误」互补）。

### 3.4 可观测性

- 关键路径（动态工具绑定、刷新、名称解析）应有 **Debug/Information** 级日志，便于区分「模型错」与「环境不同步」。
- 模型可见字符串与日志内容**分工**：模型侧短、可行动；日志侧可含上下文与堆栈。
- **JSON 参数绑定失败**（`JsonException`）与 **一般工具异常**一致：返回带 `[参数绑定失败]` 的 tool result，并打 **Warning** 日志保留完整异常。

### 3.5 动态工具与 `RunSinglePass`

- 主会话 MAF **不再**在「首轮零工具」时由服务端自动追加一轮全量工具续跑；依赖动态工具路径（`search_available_tools` / `activate_tools`）与提示词约束模型自行接地。
- 动态工具下 `activate_tools` 成功后若已原地刷新 `ChatOptions.Tools`，会清除 `ExpansionOccurredInLastPass`，避免再开一层仅因「曾激活」而多跑的 `RunSinglePass`（否则新 Agent 会话常从头再跑浏览器步骤）。
- **首轮 UserSkill**：`contextWindow.dynamicTooling` 中 `bootstrapUserSkillIds`（SKILL 的 `name`/技能 Id，可多行或列表）与 `bootstrapIncludeAllEnabledUserSkills` 可将对应 UserSkill 并入首轮 bootstrap，仍受 `ClientTypeToolFilter` 约束；与 `search`/`activate` 互补（见 `SessionToolResolver.GetDynamicBootstrapTools`、`DynamicToolingTurnState.BootstrapFunctionNamesOrder`）。

### 3.6 内置完成度评判与一次 MAF 续跑（实验性）

- **用户不可配置**：无单独子 Agent / 模型条目；实现为 `BuiltinTurnCompletionVerifier`，与主会话共用 `IChatClient`（主模型），**无工具**、短输出 JSON。
- **触发条件（代码硬编码）**：动态工具开启、strip reasoning 后的**可见助手正文为空**、且本轮曾发生 `search_available_tools` 或 `activate_tools` 调用。**不**根据 reasoning 流条数或正文触发（与 harness-engineering 一致）；评判输入**不含** reasoning 全文。
- **行为**：主 MAF pass 结束后调用评判；若解析到 `{"complete":false,...}`，向前端发一条 `stream_warning`，向 history 追加内置续跑 user 提示，再跑 **至多一轮** `EnumerateStreamingAsync`（同一 `ToolsForAgentRound` / `DynamicToolingState`）；**不在续跑后再次评判**（避免套娃）。评判超时、解析失败或异常 → 不续跑。
- **代码索引**：`backend/Services/BuiltinTurnCompletionVerifier.cs`，`ChatService` 主会话 MAF 分支。

### 3.7 WebSocket 时间线 chunk（`blockSeq` / `blockKind`）

- 主会话流式中 **`reasoning_chunk` 与 `stream_chunk`** 每条须带 **`blockSeq`**（非负整数，同一段多 chunk 共用）与 **`blockKind`**（`think` | `answer`），由 `TimelineBlockStreamCoordinator` 在 `stream_start` 后维护；**`tool_invocation_start`** 时清空活动段类型以便工具后续新开段。多端 UI 按 `blockSeq` 排序挂载逻辑段。

### 3.8 已知局限（评审代码时核对）

- **配置 `UseAgentGroupChatMainSession` 时**：`MafAgentGroupChatSessionRunner` 中 Worker 的工具列表来自 `MafRuntimeToolFacade.GetToolsForSession`，**未**传入主路径里 `FinalizeToolingPhaseTurn` 得到的 `turn.ToolsForAgentRound`（含动态 bootstrap），也**未**绑定 `ToolListMutationTarget`。若同时开启 **动态工具** 与 **Group Chat 主会话**，可能出现与主 MAF 单 Agent 路径不一致的「环境诚实」与 `activate_tools` 刷新行为，需在编排层对齐后再依赖动态工具生产使用。
- **框架层「Function 'x' not found」**（多见于请求内 `tools` 与模型 `tool_calls` 不一致）：日志可能来自 MEAI/网关而非 `ToolInvocationMiddleware`；修复依赖 **请求侧 tools 列表**与 **限定名解析**，中间件仅在进入 `next` 后捕获的执行期异常。

## 6. 可选：对模型侧的软期望（非服务端验收项）

以下依赖模型能力，**不作为后端单测门槛**，仅与提示词一起起辅助作用：

- 读到 tool result 中的错误后，修正参数或换工具重试，而不是重复相同错误调用。
- 涉及本机写操作时，以**本轮工具返回**为准向用户确认，不凭推理冒充已执行（与 `提示词清单` 中工具接地一致）。
