# 工具调用状态可见 + 错误反馈与友好文案 — 实施计划

本文档为「工具调用状态对用户可见」与「错误反馈与友好文案」两个功能的实施计划，便于按步骤落地。

---

## 一、目标与范围

| 功能 | 目标 |
|------|------|
| **工具调用状态可见** | 用户发送消息后，在 AI 流式回复过程中能看到「正在执行：xxx」「执行完成：xxx」，减少“卡住”的错觉，并知道 AI 实际调用了哪些工具。 |
| **错误反馈与友好文案** | 工具失败或整体请求失败时，前端展示明确、可操作的提示（如「请到设置页检查 API 密钥」「文件被占用，请关闭后重试」），而不是生硬的异常信息。 |

---

## 二、协议与数据格式

### 2.1 新增 WebSocket 消息类型

在现有 `WsMessage` 基础上增加以下类型（均非必填字段按需使用）：

| type | 方向 | 说明 | 建议字段 |
|------|------|------|----------|
| `tool_invocation_start` | 后端 → 前端 | 某工具即将执行 | `plugin`, `function`, `summary?`（可选，如 "读取 Excel"） |
| `tool_invocation_end` | 后端 → 前端 | 某工具执行结束 | `plugin`, `function`, `success`, `message?`（简短结果或错误说明） |
| `tool_failed` | 后端 → 前端 | 工具执行失败且希望单独强调 | `plugin`, `function`, `code?`, `message`（友好文案） |

- `tool_invocation_start` 与 `tool_invocation_end` 成对出现，便于前端显示「执行中 → 完成」。
- 若希望简化，可只做 `tool_invocation_start` / `tool_invocation_end`，在 `tool_invocation_end` 的 `success: false` 时用 `message` 承载友好错误文案，不再单独发 `tool_failed`；若希望错误更醒目，可保留 `tool_failed` 作为补充。

### 2.2 WsMessage 扩展（后端）

在 `MessageRouter.cs`（或当前定义 `WsMessage` 的文件）中为 `WsMessage` 增加可选字段，例如：

```csharp
// 工具调用状态（tool_invocation_start / tool_invocation_end / tool_failed）
[JsonPropertyName("plugin")]
public string? Plugin { get; set; }

[JsonPropertyName("function")]
public string? Function { get; set; }

[JsonPropertyName("success")]
public bool? Success { get; set; }

[JsonPropertyName("summary")]
public string? Summary { get; set; }
```

- `tool_invocation_end` 时：`Success == true` 表示成功，`false` 表示失败；`Message` 可放在现有 `Content` 或新字段 `Message` 中（需统一约定，见下）。
- 若用 `Content` 表示工具结束时的说明文案，可不再新增字段，仅用 `type` + `plugin` + `function` + `success` + `Content`。

建议约定：

- **tool_invocation_start**：`type` + `plugin` + `function`，可选 `summary`（或直接用 `Content`）。
- **tool_invocation_end**：`type` + `plugin` + `function` + `success` + `Content`（成功时为简短结果摘要，失败时为友好错误文案）。

## 三、工具调用状态可见 — 实现步骤

### 3.1 后端：新增 ToolStatusFilter

- **位置**：`backend/Filters/ToolStatusFilter.cs`（新建）。
- **职责**：在每次 Kernel 函数调用前后向当前会话推送「开始」与「结束」消息。
- **依赖**：`SessionManager`（发送 WebSocket）、`SessionContext.GetSessionId()`（当前会话）、`ILogger`。
- **逻辑**：
  1. 在 `OnFunctionInvocationAsync` 中取 `sessionId = SessionContext.GetSessionId()`；若为空则只调用 `next(context)`，不推送。
  2. 发送 `tool_invocation_start`：`plugin` = context.Function.PluginName（或类似），`function` = context.Function.Name，可选 `summary` 用一句话描述（如 "读取 Excel 区域"）。
  3. `await next(context)`。
  4. 根据 `context.Result` 或异常决定成功/失败，发送 `tool_invocation_end`：`success`、`Content` 为简短说明或友好错误文案（见第四节）。
  5. 若在 `next(context)` 外发生异常，在 catch 里发送 `tool_invocation_end(success: false)` 或 `tool_failed`，并记录日志。
- **注册**：在 `ChatService` 的 `RebuildKernelAsync` 中，将 `ToolStatusFilter` 注册到 `kernel.FunctionInvocationFilters`（建议在 `SessionContextFilter` 之后、`SecurityFilter` 之前或之后，视是否要在「安全拦截」前就展示「即将执行」而定；一般放在 SessionContext 之后即可）。

### 3.2 后端：SessionManager 注入

- `ToolStatusFilter` 需要 `SessionManager` 以调用 `SendToAsync(sessionId, json)`。
- 在 `Program.cs` 的 DI 中，`SessionManager` 应已注册；在创建 `ToolStatusFilter` 时通过构造函数注入 `SessionManager` 和 `ILogger<ToolStatusFilter>`。

### 3.3 前端：处理新消息类型

- **文件**：`chrome-extension/sidepanel.js`（以及若使用单独消息处理模块，则一并修改）。
- 在 `handleMessage` 的 `switch (msg.type)` 中增加：
  - `tool_invocation_start`：在「当前正在回复」的 assistant 气泡区域（或统一的状态区）显示一行「正在执行：{plugin}.{function}」或 `summary`（若有）；可加 loading 样式。
  - `tool_invocation_end`：将对应「正在执行」更新为「✓ 完成」或「✗ 失败」，并附带 `Content` 的简短说明；若希望失败更醒目，可再根据 `tool_failed` 或 `success === false` 高亮显示。
- **UI 建议**：
  - 在流式气泡下方或内部维护一个「本次回复过程中的工具调用列表」；每收到 `tool_invocation_start` 追加一项，收到 `tool_invocation_end` 更新该项状态与文案。
  - 或更简单：只显示「当前正在执行：xxx」，结束时清空或改为「已执行：xxx」的简短列表（避免界面过于复杂）。

### 3.4 前端：与流式回复的配合

- 当前逻辑：`stream_start` → 创建/复用流式气泡，`stream_chunk` → 追加内容，`stream_end` → 结束并可能渲染 canvas 等。
- 工具调用可能发生在两次 `stream_chunk` 之间（即模型先输出一段文字，再调用工具，再输出）。因此：
  - `tool_invocation_start` / `tool_invocation_end` 与 `stream_chunk` 会交错到达；
  - 前端应把「工具状态」与「当前 assistant 气泡」关联（例如都挂在当前正在积累内容的那条消息上），这样用户看到的是「同一轮回复里的文字 + 工具执行状态」。

## 四、错误反馈与友好文案 — 实现步骤

### 4.1 后端：统一友好文案映射

- **位置**：`backend/Services/ErrorMessageHelper.cs`（新建）或 `backend/Helpers/` 下。
- **职责**：根据异常类型或错误码返回面向用户的简短中文文案，便于在 `tool_invocation_end` 的 `Content` 和全局 `error` 消息中使用。
- **建议映射表**（可按需增删）：

| 情况 | 识别方式（示例） | 友好文案建议 |
|------|------------------|--------------|
| API 密钥无效/过期 | 401、Unauthorized、invalid api key 等 | 请到设置页检查并更新 API 密钥。 |
| 网络/请求超时 | TimeoutException、Timeout 等 | 请求超时，请检查网络后重试。 |
| 文件被占用 | IOException、文件正在使用 等 | 文件正被其他程序占用，请关闭后重试。 |
| 文件未找到 | FileNotFoundException | 未找到指定文件，请检查路径是否正确。 |
| 无权限访问 | UnauthorizedAccessException | 没有访问该文件或目录的权限。 |
| 模型/服务错误 | 5xx、Server Error 等 | 模型服务暂时不可用，请稍后重试。 |
| 默认 | 其它 | 操作失败，请稍后重试。若持续出现，请查看日志或联系支持。 |

- **接口**：例如 `string GetFriendlyMessage(Exception ex)` 或 `string GetFriendlyMessage(string errorCode, string? detail)`，在 `ToolStatusFilter` 和 `Program.cs` 的 catch 中调用。

### 4.2 后端：在 ToolStatusFilter 中使用友好文案

- 在 `tool_invocation_end`（或 `tool_failed`）时，若 `context.Result` 表示失败或捕获到异常，不直接暴露 `ex.Message`，而是调用 `ErrorMessageHelper.GetFriendlyMessage(ex)`（或类似），将返回值放入 `Content` 或 `message` 字段。
- 若工具返回的是业务错误信息（如「该命令不在白名单中」），可保留原意，仅对典型异常做映射。

### 4.3 后端：全局错误（HandleChatStream）友好化

- **文件**：`backend/Program.cs`，在 `HandleChatStream` 的 `catch` 中，当前为 `Content = $"AI 服务出错: {ex.Message}"`。
- 改为：先通过 `ErrorMessageHelper.GetFriendlyMessage(ex)` 得到友好文案，再赋值给 `Content`；若需要技术细节，可在日志中记录完整异常，前端仅展示友好文案。

### 4.4 前端：错误展示增强

- **现有**：`type === "error"` 时已用 `addBotMessage(\`⚠️ ${msg.content}\`)` 展示。
- **可做**：
  - 为错误类气泡增加固定 class（如 `message--error`），在 `sidepanel.css` 中设置醒目样式（边框/背景色/图标）。
  - 若协议中带有 `code` 或 `retryable` 等字段，可显示「重试」按钮或「前往设置」链接（如 401 时跳转 options 页）。
- **工具失败**：若使用 `tool_invocation_end(success: false, Content: 友好文案)`，在同一工具状态行显示该文案；若单独使用 `tool_failed`，可同样用 `message--error` 样式突出显示。

## 六、验收要点

- **工具调用状态**：用户说「帮我读一下 D:\test\a.xlsx 的 A1」后，在回复过程中能看到类似「正在执行：Excel.ReadExcelRange」和「✓ Excel.ReadExcelRange 完成」的提示。
- **错误反馈**：故意使用错误 API Key 或占用文件时，界面显示友好文案（如「请到设置页检查并更新 API 密钥」「文件正被其他程序占用，请关闭后重试」），而不是堆栈或原始异常信息。
- **兼容性**：未收到新消息类型的旧前端仍能正常显示流式回复与 error；新前端在未收到工具状态时行为与现在一致。

以上为「工具调用状态可见」与「错误反馈与友好文案」的完整实施计划，可按任务清单逐步实现。
