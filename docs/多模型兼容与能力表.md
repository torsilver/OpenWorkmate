# 多模型兼容困难与 Taskly 做法

> **目的**：把「为什么 OpenAI 形状还不够」「我们踩过什么坑」「仓库里现在怎么兜」写在一处，方便 onboarding、排错和后续扩展。  
> **实现以代码为准**；行为变更时请同步改本文与 [`ModelProfiles/MERGE_RULES.md`](../backend/ModelProfiles/MERGE_RULES.md)。

---

## 1. 直觉与现实的落差

很多人（包括早期本项目的假设）会认为：**只要供应商提供 OpenAI 兼容的 `POST /v1/chat/completions` + SSE，用官方 OpenAI SDK 接到 `Microsoft.Extensions.AI`（MEAI）的 `IChatClient`，再交给 MAF `ChatClientAgent`，就能通吃各家模型。**

实际上「兼容」通常只保证：

- HTTP 路径、大体上的 JSON 字段名、流式 `data:` 行形态相近；

并不保证：

- **多轮对话 + 工具调用** 时，各家对 **assistant 消息** 里是否必须带 **`reasoning_content` / thinking 块** 的校验一致；
- 同一字段在不同云上的语义（例如百炼的 `enable_thinking`、Kimi 的 `thinking`）完全一致；
- 流式阶段旁路解析出的推理片段，与下一轮请求里必须回传的「历史」是否由同一套协议约束。

因此会出现：**第一轮 200、工具也执行成功，第二轮 400**，而错误信息指向「历史里某条 assistant 缺字段」——这不是「模型 ID 写错」一类问题，而是 **协议 + 状态机** 问题。

---

## 2. 典型案例（本项目已遇到）

### 2.1 Kimi（Moonshot）类：`thinking` 与 `reasoning_content`

当上游开启 **thinking** 且历史中已有 **assistant + `tool_calls`** 时，部分实现会要求：**同一条 assistant 在后续请求里必须仍带 `reasoning_content`**。若客户端或中间层在拼下一轮请求时只保留了 `content` / `tool_calls`、丢掉了推理块，上游会直接 **`400 invalid_request_error`**（例如提示 *thinking is enabled but reasoning_content is missing…*）。

这与「模型已经决定要调工具」并不矛盾：校验的是 **API 合同**（完整 assistant 记录），不是人类语义上的「有没有必要再想一遍」。

### 2.2 阿里百炼（DashScope）OpenAI 兼容

百炼在 OpenAI 兼容模式下还有 **`enable_thinking`、`thinking_budget`、`enable_search`** 等**非 OpenAI 标准顶层字段**。本项目在 [`DashScopeChatRequestMerge`](../backend/Services/DashScope/DashScopeChatRequestMerge.cs) 中按 [`AiModelEntry`](../backend/ConfigService.cs) 合并进请求体；并对 **DashScope 域名** 的流式响应在 [`DashScopeOpenAiCompatHandler`](../backend/Services/DashScope/DashScopeOpenAiCompatHandler.cs) 做 SSE 旁路解析 `reasoning_content`（用于 UI `reasoning_chunk`，**不**写入当前实现下的会话持久化 transcript——见 [`ChatSessionPersistenceHelper`](../backend/Services/Chat/ChatSessionPersistenceHelper.cs)）。

### 2.3 网关路径（`ai-gateway`）

本地 **AI Gateway**（如 `POST /llm/v1/chat/completions`）对上游多为 **透明转发**（见 [`LlmChatCompletionsEndpoint`](../ai-gateway/Endpoints/LlmChatCompletionsEndpoint.cs)）。**兼容差异不会因为在前面加了一层网关就自动消失**；仍须在 **客户端请求体** 或 **上游可接受的参数** 上处理。运维与落盘、策略 UI、`routeMode` 等见 [ai-gateway.md](./ai-gateway.md)。

## 3. 技术栈里「谁负责什么」

| 层次 | 作用 | 不保证的事 |
|------|------|------------|
| **OpenAI SDK + MEAI `IChatClient`** | 统一调用形状、流式、工具管道接入 | 各家对历史消息、thinking 的**额外校验** |
| **MAF `ChatClientAgent`** | Agent 循环、工具中间件、会话内 history | 不替你解决「某云缺字段就 400」的根因 |
| **本项目 `ChatService` + HTTP 处理器链** | 网关头、百炼合并、按模型条目的日志与行为 | 不是「再包一层就全模型语义一致」 |

主会话 HTTP 链的大致顺序（概念上）：**模型能力合并（见下）→ 百炼专用合并/流式 tap → 日志 → 网关头 → 上游**。具体以 [`CreateDirectChatClient`](../backend/ChatService.cs) 为准。

## 4. Taskly 当前做法（能力表 + 出站补救）

### 4.1 目标与范围

- **不追求**短期内「全宇宙模型零配置」；先对 **少数目标模型** 建立**可版本化**的元数据与私有策略。
- 公开元数据基底参考 **LiteLLM** 仓库的 [`model_prices_and_context_window.json`](https://github.com/BerriAI/litellm/blob/main/model_prices_and_context_window.json)（本仓库只**剪枝摘录**到 [`backend/ModelProfiles/vendor/model_prices_excerpt.json`](../backend/ModelProfiles/vendor/model_prices_excerpt.json)，并在 [`SOURCES.md`](../backend/ModelProfiles/SOURCES.md) 中 pin 参考提交，便于 bump）。
- **私有策略**写在 [`backend/ModelProfiles/taskly-overlay.json`](../backend/ModelProfiles/taskly-overlay.json)，合并规则见 [`MERGE_RULES.md`](../backend/ModelProfiles/MERGE_RULES.md)。

### 4.2 运行时合并：`ModelProfileRegistry`

- 实现：[`ModelProfileRegistry`](../backend/Services/ModelProfiles/ModelProfileRegistry.cs)。
- 在 **配置重建运行时**（`ChatService.RebuildRuntimeAsync`）会 `Reload()`，从输出目录下的 `ModelProfiles/` 读入 vendor + overlay。
- 与具体一条用户模型配置的绑定方式：**显式** [`AiModelEntry.modelProfileKey`](../backend/ConfigService.cs)（JSON：`modelProfileKey`），与 LiteLLM 键一致（例如 `moonshot/kimi-k2.6`、`dashscope/qwen3-max`）。**不用** `modelId` 猜 profile，避免「控制台 id」与「社区表 id」不一致。

### 4.3 首版「消费」：诊断为主

重建运行时时会对已绑定 profile 的条目打日志，包括：

- 与 LiteLLM 摘录一致的 **上下文上限提示**（**不**自动写入 `ContextLength`，避免与 [`ContextWindowConfig`](../backend/ConfigService.cs) 双源冲突）；
- **`SupportsVision`** 与能力表是否不一致的 **Warning**，便于设置页与真实能力对齐。

### 4.4 Kimi 工具延续轮的务实补救：`ModelProfileChatRequestMergeHandler`

在「完整保留 `reasoning_content` 进 MEAI 历史」尚未做满之前，overlay 对 Kimi 启用了 **`suppressUpstreamThinkingWithTools`**：

- 实现：[`ModelProfileChatRequestMergeHandler`](../backend/Services/ModelProfiles/ModelProfileChatRequestMergeHandler.cs)。
- 当该模型条目绑定的 profile 要求抑制，且请求 JSON 的 `messages` 中出现 **assistant + `tool_calls` 且缺少 `reasoning_content`** 时，在请求根上写入 **`thinking: false`**，降低 **400** 概率。
- **权衡**：工具后续轮可能失去 thinking；这是产品/工程上的显式取舍，直到实现「历史原样带回 reasoning」为止。

同时 overlay 保留 **`requiresReasoningEchoWithTools`** 标记，供文档、日志与将来「真·回传 reasoning」路径联动。

## 5. 与「业界抹平杂乱」的关系

- **LiteLLM 等大 JSON / 网关**：擅长定价、上下文、部分能力布尔值与路由；**不能**替代每条产品链路里对 **thinking + 工具 + 持久化** 的专项处理。
- **MEAI / MAF**：解决 **.NET 侧抽象与 Agent 编排**；**不**保证所有供应商在工具多轮上的校验策略一致。

Taskly 的策略是：**公开表做 baseline + 私有 overlay 写清「怪癖与补救」+ HTTP 层可执行的最低成本补丁**，并逐步把「补丁」换成「协议正确」（若业务需要完整 thinking）。

---

## 6. 维护清单（给未来的自己）

1. **Bump LiteLLM 摘录**：上游键名或字段变了，先更新 `model_prices_excerpt.json` 与 `SOURCES.md` pin，再跑 `dotnet test --filter ModelProfileRegistry`。
2. **新模型**：优先增加 vendor 片段键 + overlay；在设置文档中说明应填的 `modelProfileKey`。
3. **若移除 `thinking: false` 补救**：须同步实现 **assistant 历史含 `reasoning_content`**（或上游允许关 thinking 的官方参数），并回归 **工具多轮** 场景。
