# 动态工具选择与技能选择：实现说明

本文描述主会话中「动态工具（Dynamic Tooling）」与「渐进式用户技能（Progressive User Skill）」在后端的实现方式：配置入口、索引与检索、引导工具链、与 MAF 流式会话的衔接。实现以 Chrome 等产品端一致的后端逻辑为准。

---

## 1. 总览

- **动态工具**：首轮不把「当前客户端允许的全部业务工具」塞进 `ChatOptions.Tools`，只给 **bootstrap 子集**（检索/激活元工具 + 各端保底能力等）；模型通过 `search_available_tools` 查候选、`activate_tools` 把业务工具加入本轮可用集合；必要时在同一轮 MAF 流内 **原地刷新** 工具表。
- **技能选择**：与业务工具检索 **分离**。已启用技能的 **元数据**（Id + 短描述）进入 **system 首段**；`SKILL.md` 正文与 `references/` 等 **不进入默认上下文**，由模型按需调用 `load_user_skill_instructions` 读盘。发现与排序由 `search_available_skills` / `select_skill_for_turn` 辅助。

---

## 2. 配置开关

配置位于 `AppConfig.ContextWindow.DynamicTooling`（JSON 路径约 `contextWindow.dynamicTooling`），类型为 `DynamicToolingConfig`（`backend/Services/DynamicTooling/DynamicToolingConfig.cs`）。

| 字段 | 含义 |
|------|------|
| `enabled` | 为 `false` 时退回旧行为：单次 MAF 流式 + **全量**该端允许工具。 |
| `maxOuterLoops` | 动态工具模式下 MAF **外层**循环次数上限（与 expansion 标记配合，见下文）。 |
| `maxSearchPerTurn` / `maxActivatePerTurn` | 每用户轮内 `search_available_tools` / `activate_tools` 调用次数上限。 |
| `maxSkillSearchPerTurn` / `maxSkillSelectPerTurn` | 每用户轮内 `search_available_skills` / `select_skill_for_turn` 上限。 |
| `requireSkillSelectBeforeLoad` | 为 `true` 时，`load_user_skill_instructions` 只能加载已在 `select_skill_for_turn` 中选中的技能 Id。 |
| `fallbackToFullAllowlistWhenNoActivation` | 若整轮动态流程结束后 **从未激活任何业务工具**，再跑一轮带 **全量允许列表** 的 agent（提高可用性）。 |

说明：`bootstrapUserSkillIds`、`bootstrapIncludeAllEnabledUserSkills` 已废弃，运行时会打警告并忽略；用户技能改为 **system 元数据 + `load_user_skill_instructions`**。

---

## 3. 工具阶段：何时进入动态模式

在 `ChatService.RunStreamChatToolingPhaseAsync`（`backend/ChatService.StreamPhases.cs`）中：

1. 读取 `ctxConfig.DynamicTooling`（缺省等价于 `enabled: true` 等默认）。
2. 若 `dynCfg.Enabled`：
   - 构建 `ToolCatalogIndex`：`ToolCatalogIndex.BuildFromAllowedTools(registry, clientType, sessionId)`，仅含 **当前端允许** 且 **非元工具** 的条目（元工具名见 `DynamicToolingConstants.MetaFunctionNames`）。
   - 构建 `SkillCatalogIndex`：`SkillCatalogIndex.BuildFromEnabledSkills(_skillService.GetAllSkills())`，仅 **已启用** 技能。
   - 构建 **bootstrap 工具列表**：`SessionToolResolver.GetDynamicBootstrapTools`（见下一节）。
   - 创建 `DynamicToolingTurnState`，并写入本轮 `turn.DynamicToolingState`。
   - 调用 `FinalizeToolingPhaseTurn`，其中 `appendDynamicToolingInstruction: true`，把 `DynamicToolingInstruction.Text`（及可选的 `BootstrapDirectToolsHint`）拼入 **身份后缀**，随本轮流式 history 注入模型（与端类型后缀合并）。

若 `enabled` 为 `false` 或阶段异常：使用 `SessionToolResolver.ResolveToolsByClientType(..., selectedPairs: null)` 全量允许工具，且不附加动态工具说明。

## 4. Bootstrap 工具列表里有什么

`SessionToolResolver.GetDynamicBootstrapTools`（`backend/Services/SessionToolResolver.cs`）按顺序尝试加入（去重按裸函数名）：

| 插件 | 函数 | 作用 |
|------|------|------|
| `AgentTooling` | `search_available_tools` | 在工具目录索引中关键词检索。 |
| `AgentTooling` | `activate_tools` | 将业务工具加入本轮激活集合。 |
| Office/WPS | `current_run_document_script`、`current_run_custom_document_script` | 文档侧保底脚本。 |
| Chrome | `run_builtin_page_script`、`run_custom_javascript_in_page` | 浏览器侧保底脚本。 |
| `CLI` | `run_command` | 全端保底。 |
| `UserSkillProgressive` | `search_available_skills`、`select_skill_for_turn`、`load_user_skill_instructions` | 技能链三件套。 |

若本轮绑定了计划（`planResult != null`），再通过 `MergePlanTools` 并入 Plan 四函数：`get_plan`、`update_plan`、`execute_plan_step`、`complete_plan`。

**当前完整工具列表** 由 `SessionToolResolver.BuildDynamicActiveToolList` 物化：**bootstrap ∪ `ActivatedFunctionNames` 中的业务工具**（再次过滤客户端允许与元工具排除）。

## 5. 工具检索：`ToolCatalogIndex`

文件：`backend/Services/DynamicTooling/ToolCatalogIndex.cs`。

- 索引条目：`PluginName`、`FunctionName`、工具 `Description` 摘要。
- **不收录** `search_available_tools`、`activate_tools` 及技能链三个函数名（避免自引用进「可激活业务工具」语义混淆；这些由 bootstrap 固定提供）。
- `Search(query, topK, pinnedFunctionNames)`：query 拆词，在 `plugin + function + description` 中 **不区分大小写** 计分（函数名命中加权）；无结果时回退为目录前若干项。空 query 时优先列出 **pinned**（即本轮 bootstrap 已暴露的函数名），再按字母序填满 `topK`。

---

## 6. 技能检索：`SkillCatalogIndex`

文件：`backend/Services/DynamicTooling/SkillCatalogIndex.cs`。

- 仅从 `SkillDefinition.Enabled == true` 的技能构建；描述截断到约 500 字符写入索引。
- 每条含：`SkillId`、`DisplayName`、`ShortDescription`、`SanitizedId`（`UserSkillToolNaming.SanitizeToFunctionName`）。
- `Search(query, topK, pinnedSkillIds)`：在 Id、展示名、消毒名、描述上打分；空 query 时 **已选中技能优先**（pinned），再按 Id 序填满。

---

## 7. 技能元数据进 System：`BuildProgressiveUserSkillMetadataBlock`

文件：`backend/ChatService.cs`（方法 `BuildProgressiveUserSkillMetadataBlock`），由 `GetActiveSystemPrompt` 拼到全局 system 中（在模型自带 system 与内置 guidance 之后）。

- 最多列出 **32** 个已启用技能；每条描述最多约 **200** 字符。
- 文案明确：**检索返回不是正文**；写盘/版式/流程类任务应先走技能链再动业务工具；正文与 `references/` **仅**通过 `load_user_skill_instructions` 加载。

这与工具索引互补：**发现**靠 system + `search_available_skills`；**重内容**靠 load。

## 8. 插件行为摘要

### 8.1 `AgentToolingPlugin`（`backend/Plugins/AgentToolingPlugin.cs`）

- 所有方法依赖 `DynamicToolingTurnScope.Current`；非动态模式返回「当前不在动态工具模式，忽略」。
- `search_available_tools`：递增 `SearchInvocationCount`，超 `MaxSearchPerTurn` 则拒绝；调用 `state.Catalog.Search`。
- `activate_tools`：校验次数上限；执行 **`DynamicToolingActivateSkillGate`**（见下）；解析 `Plugin.function` 或裸名，校验 `ClientTypeToolFilter`，写入 `ActivatedFunctionNames`；**不能激活**元工具名。

### 8.2 `DynamicToolingActivateSkillGate`（`backend/Services/DynamicTooling/DynamicToolingActivateSkillGate.cs`）

内置产品规则（代码常量 `DynamicToolingConstants.RequireSkillSearchBeforeActivateAfterToolSearch`，默认 `true`）：

- 若本轮 **已调用过** `search_available_tools`，且 **存在至少一个已启用用户技能**（`SkillCatalog.Entries.Count > 0`），且 **尚未调用过** `search_available_skills`，则 **拦截** `activate_tools`，提示须先至少调用一次 `search_available_skills`（query 可为空）。

不要求必须 `load`，仅要求「技能侧检索」与「工具侧检索」在存在技能时的顺序关系。

### 8.3 `UserSkillProgressivePlugin`（`backend/Plugins/UserSkillProgressivePlugin.cs`）

- `search_available_skills` / `select_skill_for_turn`：同样依赖 `DynamicToolingTurnScope`，并受每轮次数上限约束。
- `select_skill_for_turn`：把合法技能 Id 写入 `state.SelectedSkillIds`（**不读盘**）。
- `load_user_skill_instructions`：
  - 解析技能：`ResolveEnabledSkill` 支持 Id 或与 `UserSkillToolNaming.SanitizeToFunctionName(id)` 相等的输入。
  - 若 `requireSkillSelectBeforeLoad` 且当前技能 Id 不在 `SelectedSkillIds`，拒绝并说明原因。
  - 无 `relativeResourcePath` 时读 `SKILL.md` 去掉 frontmatter 后的正文，或回退到内存 `PromptTemplate`；有路径时在 `BaseDir` 下 **安全拼接**（禁止 `..` 与越界）。
  - 正文有字符上限（约 64K），附属文件有字节上限（512KB），超长截断并附说明。

## 9. 异步上下文：`DynamicToolingTurnScope`

文件：`backend/Services/DynamicTooling/DynamicToolingTurnContext.cs`。

- `DynamicToolingTurnState` 持有：配置、工具目录索引、技能目录索引、bootstrap 函数名顺序、`SelectedSkillIds`、各类调用计数、`ActivatedFunctionNames`、`ToolListMutationTarget` 等。
- `DynamicToolingTurnScope` 使用 **`AsyncLocal`**，在 MAF 主会话 runner 入口 `Push(state)`，使插件与中间件在同一会话异步流中读取 **当前轮** 状态。

---

## 10. MAF 主路径：外层循环、同 pass 内扩容

文件：`backend/Services/Maf/MafMainSessionStreamRunner.cs`。

1. 若 `dynamicTooling` 非空且 `Config.Enabled`：
   - `using (DynamicToolingTurnScope.Push(dts))` 包裹整段。
   - **外层** `for` 最多 `MaxOuterLoops` 次：每轮开始 `ClearExpansionFlag()`，用 `BuildDynamicActiveToolList` 得到当前工具列表，调用 `RunSinglePassStreamingAsync`。
   - 若本轮结束后 `ExpansionOccurredInLastPass` 仍为 `true`，则再开外层一轮（实际与 `activate_tools` 的配合见下节）。
2. 若配置了 `FallbackToFullAllowlistWhenNoActivation` 且 `HasActivatedAnyBusinessTool()` 为 `false`（激活集合里只有元工具或为空），再跑 **一轮** 全量允许工具的单 pass。

`RunSinglePassStreamingAsync` 内：

- 将本 pass 的 `List<AITool>` 赋给 `dts.ToolListMutationTarget`，供工具调用完成后刷新列表。

---

## 11. `activate_tools` 后原地刷新 `ChatOptions.Tools`

文件：`backend/Services/ToolInvocation/ToolInvocationMiddleware.cs`，方法 `TryRefreshDynamicToolingToolsAfterActivate`。

- 在每次工具 `next(...)` **成功返回后**，若函数名为 `activate_tools`：
  - 用当前 `DynamicToolingTurnState` 再次 `BuildDynamicActiveToolList`，**清空并替换** `ToolListMutationTarget` 列表内容。
  - 调用 `dts.ClearExpansionFlag()`，避免为「同 pass 内已扩容」再触发多余的外层 `RunSinglePass`（注释说明意图）。

因此：**同一轮 `RunStreamingAsync` 内**，模型在后续 turn 中即可看到新激活的业务工具；主要靠 **列表原地更新**，外层循环多为 **安全上限** 与其它潜在 expansion 路径预留。

---

## 12. 注入模型的文字引导

文件：`backend/Services/DynamicTooling/DynamicToolingInstruction.cs`。

- `DynamicToolingInstruction.Text`：描述推荐顺序（技能链 → 工具检索 → 激活 → 裸函数名 tool_calls）、与 `DynamicToolingActivateSkillGate` 一致的硬性规则摘要等。
- `BootstrapDirectToolsHint`：当 bootstrap 中已含非元工具（如各端脚本）时追加，避免模型误以为必须先 `activate` 才能调用这些已在列表中的工具。

---

## 13. 与静态选工具、计划撰写的关系

- **静态路径**：`SessionToolResolver.ResolveToolsByClientType` 仍用于非动态模式、异常回退、以及「全量保底再跑一轮」等场景。
- **计划插件等**：`PlanPlugin` 等可独立构建 `ToolCatalogIndex` 做检索类逻辑；主会话动态工具状态由 `ChatService` 工具阶段创建并传入 MAF runner。

## 15. 小结（给阅读代码的人）

1. **工具**：允许列表 → `ToolCatalogIndex` 供检索；首轮只绑 bootstrap；`activate_tools` 更新 `ActivatedFunctionNames` 并通过中间件 **同步** 到当前 pass 的 `ChatOptions.Tools` 引用列表。  
2. **技能**：system 给轻量元数据；`search_available_skills` / `select_skill_for_turn` 管理发现与选中集合；`load_user_skill_instructions` 负责读 `SKILL.md` 或附属文件进对话。  
3. **门控**：在「已工具检索 + 仍有启用技能」时，`activate_tools` 前强制至少一次 `search_available_skills`（可空 query）。  
4. **配置**：集中在 `contextWindow.dynamicTooling`；关闭 `enabled` 即回到全量工具单次流式行为。
