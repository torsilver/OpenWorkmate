# 多 Agent 五种组织模式选型指南

更新时间：2026-04-13  
核心参考：Claude 官方 2026-04-10《Multi-agent coordination patterns: Five approaches and when to use them》与 2026-01-23《Building multi-agent systems: When and how to use them》

## 这份文档回答什么问题

这份文档不只是解释 5 种多 agent 模式“是什么”，更重点回答：

- 什么时候应该选哪一种模式
- 每种模式适合解决什么结构性问题
- 哪些情况下容易误用
- 如果系统变复杂，应该如何从一种模式演进到另一种

先给结论：  
`不要从“最高级的模式”开始，而要从“最简单但足够解决当前问题的模式”开始。`

Claude 官方在 2026-01-23 明确强调，多 agent 并不是默认更好。只有当以下三类问题明显存在时，多 agent 才通常值得：

- `context pollution`：单 agent 上下文被无关信息污染
- `parallelization`：任务天然可并行
- `specialization`：不同子任务需要不同工具集或不同工作方式

如果这三类问题都不明显，通常先用单 agent 更稳。

---

## 五种模式总览

Claude 在 2026-04-10 的文章里归纳了 5 种典型多 agent 组织模式：

1. `Generator-Verifier`
2. `Orchestrator-Subagent`
3. `Agent Teams`
4. `Message Bus`
5. `Shared State`

可以把它们理解为 5 种“信息流组织方式”：

- `Generator-Verifier`：围绕“质量闭环”组织
- `Orchestrator-Subagent`：围绕“层级分工”组织
- `Agent Teams`：围绕“长期并行分工”组织
- `Message Bus`：围绕“事件流转”组织
- `Shared State`：围绕“共享知识库协作”组织

---

## 1. Generator-Verifier

## 它是什么

一个 agent 负责生成结果，另一个 agent 负责验证结果。  
如果验证不通过，验证者给出反馈，生成者继续修改，直到通过或达到最大迭代次数。

最典型的结构：

`生成 -> 校验 -> 失败反馈 -> 再生成`

Claude 官方把它视为最简单、也最常落地的多 agent 模式之一。

## 最适合什么场景

这类模式最适合：

- 输出质量要求高
- 成败标准可以比较明确地定义
- 错误成本高于多跑一轮的成本

典型例子：

- 代码生成后再跑测试
- 客服回复生成后做事实检查和品牌语气检查
- 合同/财务/合规文本生成后做规则校验
- 报告写完后做格式、引用、事实一致性检查

一句话判断：

`如果你最关心的是“别答错、别漏、别违规”，而不是“如何拆很多复杂子任务”，优先考虑 Generator-Verifier。`

## 什么时候优先选它

下面这些信号一出现，就很适合先选这种模式：

- 有明确 rubric、检查表、测试用例、规则集
- 输出物是一个“最终产物”，例如代码、回复、报告、摘要
- 生成和验证是相对可分离的两种工作
- 你不需要复杂协作，只需要“先做，再验”

特别适合做第一步升级：

- 从单 agent 升级成多 agent 时
- 想先把质量稳定住，而不是先做复杂架构时

## 什么时候不适合

不适合的核心情况有三类：

1. `验证标准说不清`
如果 verifier 只能说“看起来还行”，那它基本起不到真正的质量控制作用。

2. `生成和验证本身一样难`
比如非常开放的创意任务，评价“好不好”本身就高度主观，这时 verifier 很可能只能当陪跑。

3. `反馈无法推动收敛`
如果 generator 一直改不到 verifier 想要的方向，系统会反复打转，最后只是浪费 token。

## 误用风险

最常见的误用是：

- 以为“加一个 verifier”就等于质量提高

实际上 verifier 是否有用，取决于它是否有：

- 明确标准
- 可执行检查
- 具体反馈
- 停止条件

如果没有这些，系统只会制造一种“看起来经过复核”的假象。

## 设计建议

- verifier 要尽量绑定具体标准，而不是泛泛评价
- 反馈要可操作，例如“第 3 条规则未满足”，而不是“质量不够高”
- 必须设置最大迭代次数
- 失败后要有兜底策略，例如人工介入或输出最佳努力版本

## 适用度判断

最适合：

- 代码生成
- 文本合规检查
- 事实核验
- 模板化输出质量控制

一般不适合：

- 多步骤探索式研究
- 高耦合任务拆解
- 事件驱动自动化系统

## 你应该在什么情况下选它

如果你的问题本质是：

- “怎么保证产出别错？”
- “怎么让结果更可靠？”
- “怎么增加一层自动验收？”

那这通常是最先该上的模式。

## 2. Orchestrator-Subagent

## 它是什么

一个主 agent 负责整体任务理解、规划、派发和汇总。  
多个 subagent 各自处理边界明确的子任务，处理完后把结果返回给 orchestrator。

结构是：

`主控 -> 分派 -> 子任务执行 -> 汇总`

Claude Code 官方明确说它使用的就是这个模式。

## 最适合什么场景

这类模式适合：

- 总目标明确
- 子任务能清晰拆开
- 子任务之间依赖不强
- 主 agent 需要保持整体视角

典型例子：

- 代码审查：安全、测试覆盖、风格、架构分别检查
- 复杂调研：法规、竞品、技术实现分别研究
- 较大代码库分析：让子 agent 去探索不同模块，再集中汇总

一句话判断：

`如果任务可以被主 agent 拆成若干个边界清楚的小块，并且最后需要统一收口，优先选 Orchestrator-Subagent。`

## 什么时候优先选它

这是最值得优先作为“默认多 agent 起点”的模式。  
Claude 在 2026-01-23 的文章里也明确说，它是新团队进入多 agent 的好起点。

适合它的信号包括：

- 主任务复杂，但可拆为若干清晰子问题
- 各子问题只需要局部上下文
- 你不希望主 agent 被大量探索细节污染上下文
- 某些子任务可以并行做

它尤其适合解决 `context pollution`：

- 子 agent 自己在独立上下文里读大量信息
- 最后只把浓缩结果交还主 agent
- 主 agent 不必背负全部原始材料

## 什么时候不适合

不适合的核心情况：

1. `子任务高度互相依赖`
如果 A 子任务中途发现的信息需要立刻影响 B 子任务，而这种交叉很多，主控会成为瓶颈。

2. `任务不是清晰拆块，而是持续协作`
如果 worker 需要长期保留状态并在多个回合持续工作，subagent 模式就会显得太“短工制”。

3. `系统高度事件驱动`
如果流程不是主控按步骤分派，而是不断响应外部事件，这种层级模式就不自然。

## 误用风险

最常见的误用是：

- 按组织架构拆 agent，而不是按上下文边界拆

例如机械拆成：

- 计划 agent
- 执行 agent
- 复核 agent
- 总结 agent

这种拆法很容易造成：

- 信息来回传递
- 手续变多
- 每一跳都丢细节
- token 成本上升但收益不明显

Claude 官方反复强调：  
`应按 context boundary 拆，而不是按角色名拆。`

## 设计建议

- orchestrator 只保留高层目标、阶段状态、关键发现
- subagent 输出要尽量“蒸馏”，不要回传原始长文本
- 对明显可并行的任务要并行，不然只承担了多 agent 成本却没拿到速度收益
- 子任务要有明确输入、明确输出、明确结束条件

## 适用度判断

最适合：

- 代码审查
- 大型代码库探索
- 可拆解的研究分析
- 有中心决策点的复杂任务

一般不适合：

- 长期自治 worker 协作
- 高频事件流
- 强共享知识、弱中心控制的问题

## 你应该在什么情况下选它

如果你的问题本质是：

- “任务很复杂，但可以切块”
- “我需要一个总控来统一目标和最终答案”
- “我想把探索噪音隔离到子上下文里”

那这通常是最稳的选择。

## 3. Agent Teams

## 它是什么

Agent Teams 也有协调者，但 worker 不是一次性 subagent，而是持续存活、能长期积累上下文和经验的“队员”。

和 Orchestrator-Subagent 的核心区别是：

- `subagent`：做完一个子任务就结束
- `teammate`：会长期存在，接多个任务，保留自己的状态和熟悉度

结构更像：

`协调者 -> 多个长期 worker -> 持续并行推进`

## 最适合什么场景

它适合：

- 子任务彼此相对独立
- 每个子任务都不是一步完成，而是要持续多轮推进
- worker 持续保留上下文会明显提升效果

典型例子：

- 大型代码库迁移，每个服务一个 teammate
- 多个业务线并行改造
- 多个独立客户项目同时处理
- 多个长期运行的研究轨道

一句话判断：

`如果工作天然是“多条长跑赛道并行推进”，而不是“主控派几个短平快子任务”，更适合 Agent Teams。`

## 什么时候优先选它

适合它的信号：

- 每个 worker 都需要持续理解自己的领域
- 重复启动新 agent 的成本高
- 每个任务都需要多轮行动、调试、修复、验证
- 子任务之间的耦合比较低

Claude 官方在 2026-04-10 给出的典型案例是代码库迁移。  
因为每个服务都有自己的依赖、测试、配置和坑，长期保存“本地经验”会比每次从零开新 subagent 更有效。

## 什么时候不适合

不适合的核心情况：

1. `任务其实只是短子任务`
如果每个任务很短、很明确，那没必要维护长期 worker。

2. `子任务之间耦合很强`
如果队员之间经常互相影响，但又没有很好的共享机制，就容易冲突。

3. `共享资源冲突频繁`
多个 teammate 如果同时改同一套代码、表结构、文件，很容易打架。

## 误用风险

最常见的问题：

- 以为长期 worker 一定比短 subagent 更高级

实际上它的代价很高：

- 状态管理更复杂
- 完成检测更复杂
- 共享资源冲突更多
- 故障恢复更难

如果工作本身不是长期独立推进，这种模式只会徒增复杂度。

## 设计建议

- 每个 teammate 要有清晰“所有权范围”
- 尽量按服务、模块、客户、队列分区，而不要让多个队员抢同一资源
- 要有任务领取、心跳、超时、回收、冲突解决机制
- 协调者要能处理“有人两分钟完成，有人二十分钟完成”的异步现实

## 适用度判断

最适合：

- 服务迁移
- 多模块长期改造
- 大规模批处理任务
- 长时并行工程工作

一般不适合：

- 一次性分析
- 需要频繁共享中间发现的任务
- 中心化强控制流程

## 你应该在什么情况下选它

如果你的问题本质是：

- “我不是要拆一次任务，我是要长期养一组专长 worker”
- “每个 worker 都会在自己的地盘越做越熟”

那可以考虑 Agent Teams。

## 4. Message Bus

## 它是什么

各 agent 不直接彼此点对点协调，而是通过一个消息总线做 `publish / subscribe`。

也就是：

- agent A 发布事件
- 总线根据 topic / route 分发
- 订阅相应事件的 agent 接活处理

结构是：

`事件产生 -> 总线分发 -> 多 agent 消费 -> 新事件继续流转`

## 最适合什么场景

它特别适合：

- 系统由事件驱动
- 流程不是固定步骤，而是随着事件自然展开
- agent 类型会越来越多
- 不同 agent 团队需要相对独立开发和部署

Claude 官方举的典型例子是安全运营自动化：

- 告警进入
- 分诊 agent 分类
- 不同调查 agent 跟进
- 富化 agent 补上下文
- 响应协调 agent 决定动作

一句话判断：

`如果你的系统更像“事件流处理网络”，而不是“主控拆任务”，优先考虑 Message Bus。`

## 什么时候优先选它

适合它的信号：

- 有持续不断流入的事件
- 不同事件类型走不同路径
- 流程经常变化，未来还会不断接入新 agent
- 你希望 agent 之间低耦合、松耦合扩展

## 什么时候不适合

不适合的核心情况：

1. `你更需要清晰可追踪的线性流程`
总线模式很灵活，但追踪起来更难。

2. `你的系统规模还很小`
如果只有 2 到 3 个 agent，直接 orchestrator 往往更简单。

3. `路由精度不稳定`
如果路由错误会导致严重后果，而你又没有很强的可观测性和补偿机制，风险会比较高。

## 误用风险

最常见的问题：

- 以为“解耦”天然就是优点

实际上在 message bus 下，调试会明显变难：

- 一个事件可能触发连锁反应
- 你很难直观看到“系统为什么这么做”
- 一旦 router 丢事件、误分发，系统可能静默失败

Claude 官方明确提到，这类系统如果路由错误，常见问题不是“崩溃”，而是“悄悄没处理”。

## 设计建议

- 事件 schema 要稳定
- 每类事件都要有可追踪 ID
- 要做全链路日志、审计、重放
- 路由规则不要过度依赖模糊 LLM 判断
- 需要死信队列、重试和幂等设计

## 适用度判断

最适合：

- 安全告警自动化
- 工单流转
- 复杂业务事件处理
- agent 生态会持续增长的平台型系统

一般不适合：

- 小型项目
- 一次性任务
- 强需要人工可解释流程的系统

## 你应该在什么情况下选它

如果你的问题本质是：

- “持续有各种事件进来”
- “不同事件需要不同 agent 接手”
- “未来会不断新增 agent 类型”

那 Message Bus 很可能比层级式编排更自然。

## 5. Shared State

## 它是什么

多个 agent 通过共享的数据库、文档、文件系统或知识库来协作。  
没有强中心 orchestrator，或者中心作用非常弱。

结构是：

- 每个 agent 读共享状态
- 做自己的判断和行动
- 把新发现写回共享状态
- 其他 agent 再继续基于这些新发现工作

这是一种“通过共享世界状态协作”的方式。

## 最适合什么场景

它适合：

- 不同 agent 的发现会相互启发
- 共享知识比中心控制更重要
- 任务是探索式、协作式、研究式

Claude 官方给出的典型例子是研究综合系统：

- 一个 agent 查论文
- 一个查行业报告
- 一个查专利
- 一个看新闻

某个 agent 的发现会立刻影响其他 agent 的搜索方向。  
在这种情况下，让信息直接进入共享知识库，通常比所有内容都绕经 orchestrator 更高效。

一句话判断：

`如果任务本质是“大家共同建设一个不断生长的知识空间”，Shared State 通常比层级分派更适合。`

## 什么时候优先选它

适合它的信号：

- 中间发现之间强相关
- 你希望 agent 能随时看到彼此最新结论
- 你不想让 orchestrator 成为信息中转瓶颈
- 系统更像协作研究，而不是任务流水线

## 什么时候不适合

不适合的核心情况：

1. `任务边界需要强控制`
没有中心调度时，系统行为会更涌现、更难预测。

2. `重复工作代价高`
多个 agent 可能追同一条线索，造成重复劳动。

3. `没有明确终止机制`
这类系统最怕“大家一直互相响应”，但没有真正收敛。

Claude 官方特别提醒了一类问题：

- `reactive loops`

也就是：

- A 写了一个发现
- B 读到后继续写
- A 又读到 B 的回应继续写
- 系统不断消耗 token，但没有真正逼近答案

## 误用风险

最常见的问题：

- 把“共享状态”当成“自然会协作得更好”

实际上没有显式协调时，通常会出现：

- 重复研究
- 相互矛盾
- 并发写冲突
- 很难判断何时应该结束

## 设计建议

- 共享状态要有版本、锁、分区或去重机制
- 必须定义终止条件，例如时间预算、收敛阈值、指定 agent 判停
- 明确哪些字段是事实、哪些是猜测、哪些是待验证
- 共享知识最好是结构化的，而不是全靠自由文本堆积

## 适用度判断

最适合：

- 研究系统
- 情报综合分析
- 协作式知识构建
- 多源信息持续融合

一般不适合：

- 严格顺序执行的流程
- 强审计、强追责的自动化系统
- 对终止条件非常敏感但又难定义的任务

## 你应该在什么情况下选它

如果你的问题本质是：

- “每个 agent 的发现都会影响其他人的探索方向”
- “知识共享比层级控制更重要”

那 Shared State 是更自然的组织方式。

## 五种模式怎么选

## 一张快速判断表

| 你的核心问题 | 最优先考虑的模式 |
| --- | --- |
| 我最关心结果别错、别违规、别漏检 | `Generator-Verifier` |
| 任务复杂但能清晰拆块，需要主控统一收口 | `Orchestrator-Subagent` |
| 有很多相对独立的长任务并行推进 | `Agent Teams` |
| 系统持续接收事件，流程由事件驱动展开 | `Message Bus` |
| 多个 agent 要围绕同一个知识空间协作探索 | `Shared State` |

## 更细一点的选择逻辑

### 优先选 Generator-Verifier，如果：

- 你最关心的是质量保障
- 验收标准明确
- 任务本身不是复杂协作，而是产出一个结果再验收

### 优先选 Orchestrator-Subagent，如果：

- 你需要中心化目标控制
- 子任务边界清晰
- 你想用 subagent 隔离上下文污染

### 优先选 Agent Teams，如果：

- worker 需要长期保留领域上下文
- 子任务能长期自治推进
- 你能把资源边界切清楚

### 优先选 Message Bus，如果：

- 流程来自事件，而不是来自主控计划
- 你预计 agent 类型会持续增加
- 你需要强解耦和模块化扩展

### 优先选 Shared State，如果：

- 中间发现之间交叉启发非常重要
- 知识共享比流程控制更重要
- 你可以接受更涌现、更难预测的系统行为

---

## 一个实用决策树

你可以按这个顺序判断：

1. `我真的需要多 agent 吗？`
如果没有明显的上下文污染、并行价值、专业化需求，先别上。

2. `我主要是在解决质量问题吗？`
如果是，先看 `Generator-Verifier`。

3. `任务能否被主控清晰拆块并最终统一汇总？`
如果能，先看 `Orchestrator-Subagent`。

4. `worker 是否需要长期存活并积累领域状态？`
如果需要，看 `Agent Teams`。

5. `流程是不是事件驱动，而不是主控驱动？`
如果是，看 `Message Bus`。

6. `多个 agent 是否需要持续围绕同一知识空间协同探索？`
如果是，看 `Shared State`。

## 推荐的演进路径

Claude 官方的建议不是一上来就做复杂架构，而是按问题演进。

一个很合理的升级路径通常是：

1. `单 Agent`
2. `Generator-Verifier`
3. `Orchestrator-Subagent`
4. 再根据问题分岔到 `Agent Teams` / `Message Bus` / `Shared State`

原因很简单：

- `Generator-Verifier` 最容易验证价值
- `Orchestrator-Subagent` 最容易作为多 agent 通用起点
- 后三种模式都更强，但也都更复杂、更难调试

---

## 我给你的实际建议

如果你是自己做系统，而不是做研究型论文原型，我建议默认采用这条策略：

1. 先问自己能不能只用单 agent
2. 如果只是质量问题，先加 `Generator-Verifier`
3. 如果确实有复杂拆解和上下文污染，再上 `Orchestrator-Subagent`
4. 只有当你明确遇到长期并行、事件驱动、共享知识协作这三类结构性问题时，再考虑后三种

简单说：

- `想提高结果可靠性`：先 `Generator-Verifier`
- `想管理复杂任务拆解`：先 `Orchestrator-Subagent`
- `想做长期并行 worker`：用 `Agent Teams`
- `想做事件驱动平台`：用 `Message Bus`
- `想做协作研究网络`：用 `Shared State`

## 一句话总结

多 agent 的核心不是“多几个角色”，而是“如何组织上下文、信息流和协作边界”。  
选型时先看你要解决的是 `质量闭环`、`任务拆解`、`长期并行`、`事件流转`，还是 `共享知识协作`。
