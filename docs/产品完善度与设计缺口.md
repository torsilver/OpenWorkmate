# Office Copilot — 功能完备度与待完善设计

从「功能完备的 AI 应用」角度，对当前项目做一次盘点，并列出仍需要完善的设计与实现。

---

## 一、当前已具备的能力（简要）

| 维度 | 现状 |
|------|------|
| **通信** | WebSocket 双向通信、Session 隔离、可选 Token/Origin 校验、前端断线重连（指数退避） |
| **AI 编排** | Semantic Kernel、流式回复、多轮对话、工具自动调用（AutoInvoke） |
| **工具** | CLI（白名单）、Excel/Word（OpenXML）、Browser（高亮/笔记/截图/页面脚本）、File（截图存下载目录）、MCP 动态接入 |
| **技能** | Prompt 型 Skills（SKILL.md + JSON）、启用/停用、内置插件可停用 |
| **配置** | 统一配置 API、模型/端点/密钥、MCP 列表、脚本与命令白名单、Kernel 热重载 |
| **会话** | 按 SessionID 隔离、历史滑动窗口（MaxHistoryTurns）、空闲超时回收 |
| **安全** | CLI 命令白名单、页面脚本 ID 白名单、Word 路径解析（环境变量展开 + 下载目录） |
| **可观测** | Serilog 文件+控制台、部分插件（如 Word/File）写操作打日志、前端调试面板与历史 |

---

## 二、待完善的设计（按优先级与领域）

### 1. 安全与合规

| 项目 | 说明 | 建议 |
|------|------|------|
| **文件路径访问范围** | Word/Excel/File 当前可读写任意路径，存在越权与误删风险 | 引入「可写目录白名单」配置（如仅允许桌面、文档、下载、指定项目目录），在插件内统一做路径解析与校验 |
| **危险操作的人机确认（HITL）** | **已实现**：触发拦截时向后端发送 confirm_request，前端展示确认框，用户允许/拒绝后发送 confirm_response，后端据此继续执行或返回「用户拒绝/超时」 | 可选后续：对白名单内敏感操作做二次确认、可配置超时与总开关 |
| **敏感配置与日志** | API Key 等若落盘或进日志，存在泄露风险 | 配置存储考虑加密或使用系统凭据；日志中对 ApiKey、token 等做脱敏或完全省略 |
| **CORS / Origin** | 当前 `SetIsOriginAllowed(origin => true)` 注释为「暂时全部放行」 | 正式环境建议收紧为仅允许扩展 Origin，避免恶意网页直连本地服务 |

### 2. 可靠性与可观测性

| 项目 | 说明 | 建议 |
|------|------|------|
| **长对话上下文** | 仅滑动窗口删旧消息，无摘要压缩，长会话会丢失早期上下文且易顶满 Token | 在 TrimHistory 时对「被裁掉」的旧轮次做摘要（调用模型或本地摘要），再以一条系统/用户消息形式补回，或提供「新会话」入口 |
| **工具调用对用户不可见** | 用户看不到「正在调用 write_word_document」等状态，易误以为卡住 | 流式协议中增加 tool_invocation_start / tool_invocation_end（或 tool_result 摘要），前端展示「正在执行：xxx」「执行完成：xxx」 |
| **工具失败时的明确反馈** | 工具异常仅回传给模型，用户依赖模型转述，可能被弱化或遗漏 | 关键工具失败时，除返回模型外，可额外推一条 error 或 tool_failed 类型消息，前端以醒目方式展示并可选重试 |
| **请求/会话级追踪** | 日志缺少与前端 Session、单次请求绑定的 trace id | 为每个请求生成 RequestId，在 SessionContext/日志里贯穿 Chat、Filter、Plugin，便于排查「某次对话为什么失败」 |
| **健康检查深度** | 仅有 /health 存活探测 | 增加依赖检查：LLM 端点可达性、MCP 子进程状态、配置是否已加载等，供监控与启动自检 |

### 3. 体验与反馈

| 项目 | 说明 | 建议 |
|------|------|------|
| **执行中状态** | 工具执行（尤其是 RPC、写文件）时无 loading 或进度提示 | 结合上一条「工具调用可见」，前端在收到 tool_invocation_start 时显示 loading，tool_invocation_end 时收起 |
| **错误可操作性** | 错误信息偏技术（如异常 Message），用户不知如何改 | 对常见错误（如 401、超时、文件被占用）做分类与友好文案，并给出「检查设置页 API Key」「稍后重试」等引导 |
| **模式切换反馈** | 前端切换 Assistant/Workspace 会发 mode_change，后端无明确「已切换」确认 | 可选：后端回一条 stream 或单独消息确认当前模式，或在前端根据已发送的 mode_change 做乐观展示 |
| **设置页与运行时一致** | 配置热更新后，已存在会话的 Kernel 会重建，但用户可能不知道需重试失败操作 | 设置页保存后提示「配置已生效，若当前对话异常可重新发送」；或连接状态区显示「配置已更新」 |

### 4. 性能与稳定性

| 项目 | 说明 | 建议 |
|------|------|------|
| **限流** | 无按 Session 或 IP 的请求频率限制，易被单用户或恶意请求拖垮 | 按 SessionID（或 IP）做简单限流（如每分钟 N 条消息），超出返回 429 或友好提示 |
| **大文档/大表格** | Excel/Word 大文件一次性读入内存，可能 OOM 或超时 | 大文件分块读取、流式返回或分页；或限制单次读取大小并明确在工具描述中说明 |
| **MCP 进程恢复** | MCP 子进程崩溃后，当前会话或后续请求可能持续失败 | 子进程异常退出时记录并标记该 MCP 不可用，支持「重连」或下次请求前自动重启一次，并限制重试次数 |
| **RPC 超时** | 前端 RPC 已有 60s 超时，但长时间截图等仍可能让用户等待 | 对 capture_full_page 等耗时操作，前端可显示「正在截取整页…」并在超时时提示「操作超时，请缩小页面或重试」 |

### 5. 工程化与可维护性

| 项目 | 说明 | 建议 |
|------|------|------|
| **自动化测试** | 未发现单元测试/集成测试项目 | 至少为 ChatService.StreamChatAsync、SecurityFilter、Word/File 路径解析等核心路径补单元测试；WebSocket 握手与消息流可做集成测试 |
| **API 与协议版本** | WebSocket 消息格式、REST API 无版本标识 | 消息体或 URL 增加 version 字段，便于后续兼容与灰度 |
| **配置校验** | 保存配置时未做完整校验（如 Endpoint 格式、必填项） | 保存前校验必填与格式，失败时返回具体字段错误，前端展示到表单项 |
| **依赖与漏洞** | 如 ImageSharp 等依赖存在已知漏洞提示 | 定期升级依赖并处理安全告警，或通过 Suppress 明确记录与排期 |

### 6. 产品化与扩展

| 项目 | 说明 | 建议 |
|------|------|------|
| **多用户/多设备** | 当前为单机单进程，无用户概念 | 若未来支持多用户或远程访问，需要身份与权限模型、配置与会话按用户隔离 |
| **安装与升级** | 用户可能不知道新版本或如何更新 | 扩展内提供「关于 / 检查更新」入口；后端可在 /health 或单独接口返回建议版本，前端提示升级 |
| **无障碍与 i18n** | 文案以中文为主，未系统做多语言与无障碍 | 若面向更广用户，可抽离文案键、支持多语言；关键交互保证键盘可操作与屏幕阅读器友好 |

---

## 三、建议的落地顺序（按投入与收益）

1. **高优先级（安全与体验）**  
   - 文件路径白名单（Word/Excel/File）  
   - 工具调用状态对用户可见（协议 + 前端展示）  
   - 关键错误的前端明确反馈与友好文案  

2. **中优先级（稳定性与可观测）**  
   - 对话历史摘要压缩（或「新会话」+ 文档说明）  
   - 请求级 trace id 与日志关联  
   - 按 Session 的简单限流  
   - 健康检查增强（依赖状态）  

3. **后续迭代**  
   - HITL 危险操作确认  
   - 配置加密与日志脱敏  
   - 自动化测试与 API 版本  
   - 大文件策略与 MCP 进程恢复  

---

## 四、小结

当前项目已经具备：完整的前后端通信、多工具与 MCP 扩展、Skills 与配置热更新、基础安全白名单和会话管理。从「功能完备的 AI 应用」看，主要缺口集中在：**安全边界（路径白名单、HITL）**、**可观测与反馈（工具可见、错误明确、追踪）**、**长对话与稳定性（摘要/限流/健康检查）**、以及**工程化（测试、版本、配置校验）**。按上述优先级逐步补齐，可以在不推翻现有架构的前提下，明显提升可靠性、安全性和使用体验。
