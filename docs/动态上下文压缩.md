# 动态上下文压缩（与实现对齐）

本文说明 **OpenWorkmate / OpenWorkmate 后端**里「何时压、怎么压、压完长什么样」，并与通用「触发锚点」概念对照。预算字段与裁剪顺序见 [Token预算与上下文裁剪.md](./Token预算与上下文裁剪.md)。

---

## 在工程里的两条路径（双轨）

| 路径 | 角色 | 本仓库落点 |
|------|------|------------|
| **系统侧（每用户轮前）** | 按配置与估算 token 做 MAF Compaction、轮数/token 裁剪、超长重试时再删 | `ChatService.RunStreamChatContextPhasePart1Async`、`ContextManager` |
| **模型主动** | 在换任务或已不需要旧细节时，显式压缩以释放窗口 | 插件工具 **`compact_conversation`**（`CompactConversationPlugin`，分类 Context） |

二者可同时存在：系统侧是**默认兜底**，工具是**按需加速整理**（会占用一轮工具调用）。

---

## 一、系统侧自动化（Passive / Framework）

由 **`ChatService.RunStreamChatContextPhasePart1Async`**（`ChatService.StreamPhases.cs`）在每轮流式推理前执行（与「主 Agent 是否想到要压」无关）。

### 1. Query-aware 启发式（实验，默认关）

在 MAF Compaction **之前**、且 **`CompactionQueryAwareHeuristicEnabled`** 为 true 时：若估算历史 token 已超过预算×`CompactionQueryAwareTokenPressureRatio`，则从 **`ConversationCompactBoundary`** 允许的「可删区」删除与**当前用户句**无词重叠的最旧消息（单次上限 `CompactionQueryAwareMaxRemovalsPerTurn`）。拆词与 **`CompactionRelevanceDiagnostics`** 一致，便于与 `OpenWorkmate_COMPACTION_RELEVANCE_LOG=1` 日志对照。实现见 **`CompactionQueryAwareHeuristic`**。

### 2. MAF Compaction（可开关）

**前置条件（同时满足）：**

- `historyBudget > 0`（由 `GetEffectiveMaxContextTokens()` 减去 `ReservedSystemTokens`、`ReservedToolsTokens`、`ReservedOutputTokens` 得到）
- `PassThroughContext == false`
- `SummarizationEnabled == true`
- `state.History.Count > 5`

**触发强度：** `triggerTokens = (int)(historyBudget * SummarizationTriggerRatio)`（默认比例 **0.9**，不是固定的 40%/70% 两档）。策略由 **`BuildCompactionStrategy`** 组装为管道：

1. **工具结果折叠** — `ToolResultCompactionStrategy`，触发条件为估算 token 超过 `triggerTokens`
2. **摘要** — `SummarizationCompactionStrategy`，同样按 token 超阈触发，`minimumPreservedGroups: 4`，摘要提示词要求不把摘要当作磁盘实时状态
3. **截断兜底** — `TruncationCompactionStrategy`，预算为整段 **`historyBudget`**

配置项见 `ContextWindowConfig`：`SummarizationEnabled`、`SummarizationTriggerRatio`、`SummarizationMaxSummaryChars` 等。

### 3. 轮数与 Token 裁剪（始终可与 Compaction 叠加）

- **`ContextManager.TrimHistory`**：`SessionConfig.MaxHistoryTurns` 限制消息条数；再按估算 token 与预算删旧消息。
- **删除顺序**：通过 **`ConversationCompactBoundary.GetFirstRemovableChatIndex`** 决定「从哪一条开始删」，避免删掉带 **`[compact_boundary:…]`** 的摘要块及其锚点消息（与 harness 规则一致：压缩边界不可随意破坏）。

### 4. 「大单次工具结果」类需求（本仓库现状）

- 管道里的 **工具结果折叠** 会按 **整段历史估算 token** 与 **`triggerTokens`** 比较，**不是**「单次工具返回 > N token 就单独结构化蒸馏」的独立锚点。
- 另有 **`TruncateToolArgsThresholdRatio` / `TruncateToolArgsKeepMessages` / `TruncateToolArgsMaxChars`**：在历史 token 占比过高时，对**保留窗口之外**的旧消息做大段截断（偏「软清理」，详见配置注释）。

### 5. 上下文长度报错时的重试

- **`ContextLengthRetryEnabled`** 等：命中上下文长度类错误时 **`ContextManager.TrimHistoryForRetry`** 与 Compaction 边界一致，先按轮数再按半预算删。

### 6. 概念性锚点（当前未作为独立后台策略实现）

以下在白皮书式方案里常见，**本仓库尚未**做成单独的「监测 Excel→Word 就清上下文」「同一工具连错 3 次就进入诊断模式」等后台任务；若未来要做，需要单独设计观测点与测试，避免与 `ConversationCompactBoundary`、推理流语义冲突（参见 `.cursor/rules/harness-engineering.mdc`）。

## 二、主 Agent 主动（Active / Tool）

**工具名：`compact_conversation`**

- 实现：**`CompactConversationPlugin.CompactConversationAsync`** → **`ChatService.CompactConversationAsync`**
- 语义：将当前会话**最旧若干轮**合并为摘要（同样走 MAF `CompactionProvider`；工具路径上 `triggerTokens`/`budgetTokens` 传 0，由策略内部行为决定）
- 适用：对话已经很长、要开始全新任务、或已给出最终结论不再需要旧细节时

**未提供**名为 `optimize_memory` / `update_project_state` / `compress_data` 的同类工具；长期记忆与注入由 **`MessageAIContextProvider`** 等路径负责，与「对话历史Compaction」分工不同。

---

## 三、压缩后的历史长什么样（回填形态）

系统摘要与工具压缩写入的历史，会遵循 **`ConversationCompactBoundary`** 约定，而非泛化的 `SYSTEM_MANAGED_STATE` 模板：

- 摘要前缀：**`[此前对话摘要]`**
- 说明文：提醒摘要**不能单独当作本机 Office 文件当前状态**
- 正文包在 **`<prior_conversation_summary>…</prior_conversation_summary>`** 内
- 单行锚点：**`[compact_boundary:ISO8601]`** — 裁剪时必须保留该标记语义，**勿删**（见 `docs/提示词清单.md` H 节与代码注释）

其后仍接**最近若干轮**完整对话；具体保留轮次由 Compaction 策略与 `TrimHistory` 共同决定。

## 四、观测与实验

- 环境变量 **`OpenWorkmate_COMPACTION_RELEVANCE_LOG=1`**：`CompactionRelevanceDiagnostics` 在压缩前打相关性诊断日志（不改变历史）。
- 环境变量 **`OpenWorkmate_CONTEXT_SNAPSHOT_DIR`**：可对送入 MAF 前的历史写脱敏快照（`ContextTurnSnapshot`）。
- 前端/trace：压缩成功时可有「历史压缩：…条消息已整理」类 trace（见 `AgentTraceFormatter.BuildContextSummarizationSuccessTrace`）。

## 五、对比小结

| 维度 | 系统侧（Part1 + Trim） | 工具 `compact_conversation` |
|------|-------------------------|-----------------------------|
| 触发 | 每轮、由配置与 token/条数门限决定 | 模型主动调用 |
| 模型额外轮次 | 无（摘要可能内部调用模型，对用户轮次透明） | 占用一次工具轮次 |
| 典型用途 | 防溢出、常规折叠与摘要 | 任务切换、主动腾窗口 |

**建议**：以 **`ContextWindowConfig` + `SessionConfig`** 调通系统侧行为；需要时再在提示中引导使用 **`compact_conversation`**，而不是假设存在未实现的 `compress_data`。
