# Taskly Windows 手工打包说明

面向：**在 Windows 上从源码打出 `stage`，并默认生成 MSI**（与 `scripts/Pack-Taskly.ps1` 行为一致）。

## 1. 构建机先决条件

| 项目 | 用途 |
|------|------|
| **.NET SDK** | 与仓库一致（当前为 .NET 10），用于 `dotnet publish` |
| **Node.js + npm** | `wps-addin-new` 执行 `npm run build` |
| **WiX CLI**（默认会打 MSI） | 终端可执行 `wix --version`。安装：`dotnet tool install --global wix`，装完**新开终端**。不要 MSI 时脚本加 `-BuildInstaller:$false` |
| **OpenSSL**（可选） | 首次打 Chrome CRX 且仓库无 `keys/taskly-chrome-extension.pem` 时，脚本可能依赖本机 OpenSSL 生成密钥 |

## 2. 推荐：一键打包（默认含 MSI）

在**仓库根目录** `d:\CodeBase\_Taskly`（或你的克隆路径）打开 PowerShell：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\windows-packaging\scripts\Pack-Taskly.ps1
```

仅打 `stage`、**不要** MSI（本机未装 WiX 或只调试中间产物）时：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\windows-packaging\scripts\Pack-Taskly.ps1 -BuildInstaller:$false
```

项目里容易混在一起的是两件事，请分开理解：

| 维度 | 选项 | 含义 |
|------|------|------|
| **目标框架（TFM）** | **`net10.0-windows`（默认）** | 含 WinForms 托盘、单实例互斥、`OfficeLegacyComInterop`、Word/Excel Interop 等；**桌面/MSI 安装场景请用这个。** |
| | **`net10.0`** | 同一套 .NET 10，但**编译时排除**上述 Windows 专用源码；无托盘、无 Office COM 插件路径；适合只要 Web/API、不要托盘与互操作的发布。 |
| **部署方式** | **默认（不加开关）** | **`--self-contained true`**：自带 .NET 运行时，体积大，目标机**不必**预装运行时。 |
| | **`-FrameworkDependent`** | **`--self-contained false`**：依赖目标机已安装对应 .NET 运行时，安装包更小。 |

说明：**`-FrameworkDependent` 不是「从 net10.0-windows 换成 net10.0」**；它只控制自包含与否。要选「纯 net10 控制台构建」请用脚本参数 **`-BackendTargetFramework net10.0`**。

`Taskly.StaticHost` 为单一 TFM；只有 AI 后台是 **双 TFM**。用户 MSI **不包含**遥测中继 `ai-gateway`（由运营侧单独部署）。

### 常用参数

| 参数 | 说明 |
|------|------|
| `-BuildInstaller` | 默认 **`$true`**（生成 **`windows-packaging\dist\Taskly.msi`**）；**`-BuildInstaller:$false`** 跳过 WiX |
| `-BackendTargetFramework net10.0-windows` | AI 后台目标框架，**默认**；含托盘与 Office 互操作。 |
| `-BackendTargetFramework net10.0` | AI 后台改为 **net10.0** 构建（无托盘/COM 互操作）。 |
| `-FrameworkDependent` | 改为框架依赖发布（体积小，目标机需装 .NET 运行时）；**默认是自包含** |
| `-SkipChrome` | 跳过 CRX / `updates.xml`（调试时可加快） |
| `-StaticHostPort 3000` | 写入 manifest / Chrome 更新 URL 的 HTTPS 端口，**默认 3000** |
| `-Configuration Release` | 构建配置，默认 `Release` |
| `-Runtime win-x64` | RID，默认 `win-x64` |

### 产物位置

- **安装包**：`windows-packaging\dist\Taskly.msi`
- **展开目录（未封装进 MSI 也可直接跑）**：`windows-packaging\stage\`
- **本机试跑**：`powershell -NoProfile -ExecutionPolicy Bypass -File .\windows-packaging\stage\Start-Taskly.ps1`

## 3. 手工分步（与脚本等价思路）

若不想用一键脚本，可按顺序自己做（路径按你的仓库根调整）：

1. **发布 AI 后台**（二选一，与 `Pack-Taskly.ps1` 的 `-BackendTargetFramework` 一致）  
   - 桌面/完整功能（默认）：  
     `dotnet publish backend\OfficeCopilot.Server.csproj -c Release -f net10.0-windows -r win-x64 --self-contained true -o windows-packaging\stage\OfficeCopilot.Server`  
   - 无托盘、无 Office COM 互操作：  
     `dotnet publish backend\OfficeCopilot.Server.csproj -c Release -f net10.0 -r win-x64 --self-contained true -o windows-packaging\stage\OfficeCopilot.Server`  
   若要与「依赖本机运行时」一致，把 `--self-contained true` 改为 `false`（并加 `-FrameworkDependent` 时脚本行为一致）。

2. **发布 StaticHost**  
   `dotnet publish windows-packaging\runtime\Taskly.StaticHost\Taskly.StaticHost.csproj -c Release -r win-x64 --self-contained true -o windows-packaging\stage\Taskly.StaticHost`

3. **Office 静态文件**   
   将 `office-addin\`（排除 `node_modules`）复制到 `windows-packaging\stage\office-addin\`

4. **WPS**   
   在 `wps-addin-new\` 执行 `npm install`、`npm run build`，将 `dist\` 复制到 `windows-packaging\stage\wps-addin\`

5. **Chrome CRX**（可跳过则自行保证 `stage\taskly-chrome` 结构）   
   `powershell -NoProfile -ExecutionPolicy Bypass -File .\windows-packaging\scripts\Build-ChromeCrx.ps1`

6. **生成 manifest / appsettings / 策略提示等**   
   一键脚本里在「Generate Office manifest…」段完成；手工时需自行对齐 `Pack-Taskly.ps1` 中的模板替换与 `Start-Taskly.ps1` 生成逻辑。

**结论**：日常维护仍以 **`Pack-Taskly.ps1`** 为准，避免漏步骤。

## 4. 仅重编 MSI（stage 已准备好）

当 `windows-packaging\stage\` 已是最新、只想改安装器或重链 MSI 时，在**仓库根**执行（路径与 `Pack-Taskly.ps1` 一致）：

```powershell
$root = "d:\CodeBase\_Taskly"   # 改成你的仓库根
wix build "$root\windows-packaging\installer\Taskly.wxs" "$root\windows-packaging\installer\WixUI_Taskly_InstallDir.wxs" "$root\windows-packaging\installer\Taskly-Dialogs.wxs" `
  -ext WixToolset.UI.wixext `
  -bindpath "stage=$root\windows-packaging\stage" `
  -bindpath "launcher=$root\windows-packaging\installer\launchers" `
  -arch x64 `
  -out "$root\windows-packaging\dist\Taskly.msi"
```

说明：

- **必须**加 `-ext WixToolset.UI.wixext`（向导 UI）。
- **必须**加 `launcher=` bindpath（`LaunchTaskly.cmd` 等安装器附属文件）。

## 5. 安装与使用提示（简）

- MSI 为 **每台计算机 (perMachine)**，一般需**管理员**安装。
- 安装完成后开始菜单有 **Start Taskly**、**Taskly client setup**；Chrome 策略见安装目录内 `Chrome-policy-hint.txt` / `Install-ChromePolicy.ps1`。
- 本机 HTTPS 若浏览器不信任：`dotnet dev-certs https --trust`。

## 6. 常见问题

| 现象 | 处理 |
|------|------|
| `wix` 不是内部或外部命令 | 已装全局工具后**新开终端**，或检查 `PATH` 是否包含 `%USERPROFILE%\.dotnet\tools` |
| WiX 报找不到扩展 | 确保 WiX CLI 与扩展版本匹配；本仓库脚本使用扩展名 `WixToolset.UI.wixext`（由 `wix build -ext` 解析） |
| 打 MSI 报代码页 / 中文相关错误 | `Taskly.wxs` 中 `Package` 已设 `Codepage="936"`，勿随意改回默认 1252 又写中文 UI 字符串 |

更完整的架构说明见同目录 **`README.md`**。
